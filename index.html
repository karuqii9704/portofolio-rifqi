02 · Incident review

When the video worked, but everything around it failed.

An anonymised remote-client engagement. No client source, identifiers, or private logs are reproduced here.

Flutter · Firebase · live streaming

Separating a cloud blocker from app-level defects

Live video playback succeeded, while comments, likes, gifts, and viewer synchronisation failed. Treating every symptom as one UI bug would have produced risky changes without fixing the system.

  1. 01

    Reproduce

    Confirmed the split behaviour: stream transport worked; Firestore-backed interaction paths did not.

  2. 02

    Read the evidence

    Traced device output and repeated failures rather than inferring the cause from UI symptoms.

  3. 03

    Identify the boundary

    Logs showed the Firestore API was disabled for the client’s cloud project. The root blocker was environment configuration—not security rules.

  4. 04

    Separate the fixes

    Documented the external blocker independently, then corrected app-owned issues including a Firestore write triggered from the render path.

  5. 05

    Verify honestly

    Moved the write after rendering with deduplication and freshness guards; application analysis returned zero errors and the APK built successfully.

“A useful debugging result is not just a patch. It is a clear boundary between what failed, why it failed, what I changed, and what still depends on someone else.”

Engineering principle used on this engagement

03 · Working system

Claude Code is part of the workflow—not the evidence.

I use AI agents daily to increase coverage and speed. Final confidence still comes from traces, diffs, tests, builds, and human review.

  1. 1

    Observe

    Reproduce the problem and collect the smallest useful trace, log, or failing test.

  2. 2

    Decompose

    Split independent research, frontend, backend, data, and QA work only where parallelism helps.

  3. 3

    Implement

    Keep ownership boundaries explicit and make the smallest change that addresses the supported hypothesis.

  4. 4

    Challenge

    Use a QA/security pass to test assumptions, error paths, permissions, and regression risk.

  5. 5

    Verify and record

    Run tests and builds, inspect the user path, and leave a concise handoff with remaining risks.
